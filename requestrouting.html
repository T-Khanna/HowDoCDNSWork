Request Routing Systems


The above image(1) shows the general reqeust routing system. The main part of the system that differs between content distribution networks is the selection algorithm. As implied by the name, this decides which surrogate server the request should be directed to. There are many different types of techniques used for implementing the sorting algorithm. In general they fall into one of two general schemes:

  1. Active Schemes (with per-request monitoring).
  2. Passive Schemes (with routing tables).

Active Scheme Routing


In Active Schemes, with active request routing, each initial end user request for a live or VOD stream is sent to a pool of request routers. These routers analyse each individual request. Over a dozen parameters are checked:

  • Where are you?
  • Does this content exist?
  • Which delivery node has the best score to deliver this content for you?
These examples of parameters are only to name a few. All these parameters are checked within the space of a millisecond.(2) This form of routing is not very common in Internet Architecture as it is less efficient and provides no significant benefit when compared to Passive Scheme Routing.

Passive Scheme Routing


Passive Schemes involve using a table which maps each request sent by a user to the appropriate surrogate server. There is no evaluation of the parameters mentioned in active schemes. An example of the mappings in a routing table is shown below.(3)

Client Location Surrogate Server
America New York
Europe Paris
Africa Paris
Asia New Delhi

So, in this example, if the client location was America, the mapping in the passive scheme table would choose the surrogate server in New York.

In the context of the Internet, requests are identified by their IP addresses. As such, this scheme is favoured over the active scheme for the Internet. Trying to implement an active web-based CDN introduces a large number of parameters to think about, especially if the parameters have to be able to cover the internet. As such, the drect mapping from client location to surrogate server is preferred as it reduces the complexity of the system.
